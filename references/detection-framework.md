# Pareto Startup Thinking Framework

## Goal

Find the smallest set of actions, bets, or decisions most likely to move a startup outcome under real constraints.

The point is not to sound focused. The point is to improve the actual bottleneck.

## Core Sequence

Use this sequence in order:

1. `Bottleneck`
2. `Leverage`
3. `Learning speed`
4. `Carry-cost`
5. `Kill or defer`

If the answer skips the bottleneck, it is usually already drifting.

## Ranking Order

Use this order:

1. Bottleneck relevance
2. Expected impact
3. Cost of delay
4. Reversibility and learning speed
5. Carry-cost added

If runway is short, move `learning speed` up even more.

## Required Questions

Before prioritizing, define:

- the outcome
- the time horizon
- the current bottleneck
- what `good enough` means right now

Without those, prioritization turns into random trimming.

## Leverage Rubric

For each candidate, judge:

- `impact`
- `confidence`
- `effort`
- `cost-of-delay`
- `carry-cost`
- `bottleneck-fit`

You do not need fake numeric precision. Clear relative judgment is enough.

## Conflict Resolver

Use these tie-breakers when the answer is not obvious:

- If one move improves the bottleneck directly and another is more elegant, pick the bottleneck move.
- If one move creates fast learning and the other creates more scope, prefer fast learning.
- If one move has uncertain upside but high build cost, choose `test first`.
- If one move lowers carry-cost on a path you already know matters, it can beat a flashier feature.
- If one move is only attractive because the team already started it, discount sunk cost.

## False-Focus Checks

Reject or down-rank work when:

- it looks important but does not touch the bottleneck
- it improves a local metric while missing the real outcome
- it adds complexity that future work will have to carry
- it is easy to talk about but not load-bearing
- it is being prioritized mainly because it is visible or already in motion

## When Pareto Is The Wrong Tool

Do not force this method when:

- the user still needs idea generation, not selection
- the biggest risk is legal, safety, or security and must be addressed regardless of leverage optics
- there is no clear outcome or time horizon
- the work is long-term platform maintenance that cannot be reduced to a short-term bottleneck bet

In those cases, say so and use a better frame.

## Action Types

For each high-leverage item, choose one:

- `do now`
- `test first`
- `measure first`
- `defer`
- `delete`
- `simplify`

## PM-Friendly Default

A good answer usually says:

- what matters most
- why it matters most
- what to stop doing
- what the next smallest move is
- why the rejected work lost

## Quick Comparison

- `RICE / ICE` help score options. This skill is better when the core issue is the real bottleneck, not the scoring formula.
- `Theory of Constraints` helps find the bottleneck. This skill adds startup-specific learning speed and carry-cost.
- `Lean Startup` helps test uncertainty. This skill uses that lens after identifying the vital few.
